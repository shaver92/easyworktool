## Problem Statement

构建一个家庭日常记账工具，支持多家庭成员独立记账、自定义费用类型、多维度统计（周/月/年）、个人预算上限及超支预警。

核心需求：
1. 多身份登录（家庭成员：爸爸、妈妈、子女等）
2. 自定义费用类型
3. 周/月/年多维度统计
4. 个人支出上限设置 + 预警
5. 方便部署，线上使用

## What Makes This Cool

**飞书Bot作为录入入口，Streamlit作为分析看板。** 家人在飞书群里直接发消息记录消费（"买菜 128"），Bot自动解析并入库。每周一早上Bot推送家庭消费周报到群聊。Streamlit页面不做日常录入——只做深度图表分析和配置管理。

录入摩擦降到零。洞察触手可及。

## Constraints

- 家庭规模：6-8人，日消费记录几十条
- 数据存储：SQLite（WAL模式，支持轻度并发写入）
- 部署：轻量级 VPS，两个进程（FastAPI + Streamlit）共享一个 SQLite 文件
- 非商业化：无付费/订阅需求
- 飞书依赖：Bot录入需飞书账号；非飞书成员通过Streamlit网页fallback录入

## Data Model

```
users
  id, display_name (爸爸/妈妈/...), feishu_open_id (nullable),
  auth_method (feishu_oauth | web_pin), web_pin_hash (nullable),
  is_admin (bool), created_at

categories
  id, name (餐饮/交通/教育/...), icon (emoji), created_by (user_id),
  is_system (bool, 预设分类不可删)

expenses
  id, user_id, category_id, amount (正数=支出, 负数=退款),
  type (expense | refund), note, recorded_at, source (feishu_bot | web)

budgets
  id, user_id, category_id (nullable, null=总预算),
  amount (月度上限), month (YYYY-MM),
  warn_threshold (float, 0.8=花到80%时提醒)
```

关键设计决策：
- **隐私模型**：个人消费明细仅本人和管理员可见；家庭汇总（总额、分类合计，不显示个人明细）所有成员可见
- **退款处理**：amount 存负数，type=refund，统计时自动扣减
- **认证分层**：飞书用户走 OAuth 自动识别，非飞书用户用 4 位 PIN 码 + 下拉选择身份
- **SQLite WAL**：启用 WAL 模式，允许多个读操作与一个写操作并发，家庭场景足够

## Message Format (飞书Bot)

支持三种输入格式，正则依次匹配：

```
格式1: 金额 类别 备注     →  "128 餐饮 中午请客"
格式2: 金额 类别          →  "36.5 交通"
格式3: 金额 备注          →  "32 早饭"  （类别=未分类，随后手动归类）
```

解析失败时Bot回复示例："没识别出来，试试这样：`128 餐饮 中午请客`"

MVP阶段只支持严格格式（正则）。LLM模糊解析作为Phase 2，在正则覆盖率低于80%时接入。

## Premises

1. SQLite WAL模式足够承载家庭级数据量，无需引入外部数据库
2. 多身份本质是 user_id + display_name，底层是通用多用户模型。预设角色名仅作为 display_name 的快捷选项
3. Streamlit结合飞书OAuth做身份识别；非飞书成员用PIN码+选择器登录Streamlit
4. 预算预警MVP阶段在Streamlit页面红色高亮提示；Phase 2通过飞书Bot推送超支通知
5. 自定义费用类型和多维度统计是核心差异化，不是附加功能
6. 飞书Bot做录入，Streamlit做分析+配置+预警，职责边界清晰；Bot推送通知属于Phase 2

## Cross-Model Perspective

独立AI顾问的冷读：

1. **最酷版本：** 飞书Bot作为主录入通道。发"买菜 128"到Bot即完成记账，Bot自动分类。Streamlit退居二线成为管理后台。
2. **核心洞察：** "多维度统计：周、月、年"才是情感钩子——不是录入，不是预算，是*模式识别*。用户想在周末翻图表，发现"这个月外出吃饭比上个月多了40%"。先做Dashboard，后做表单。
3. **参考项目：** Actual Budget（轻量，本地优先，API好）——借鉴其数据模型设计；Firefly III过重，不引入。
4. **周末原型：** 单行消费录入 + "本周"仪表盘。录入"早饭 32"即保存。周日：本周vs上周分类对比图。

## Approaches Considered

### Approach A: Streamlit 全栈
纯Web端录入+展示。Streamlit表单录入消费记录，echarts/pandas做统计图表，自定义session管理，SQLite存储。部署到Streamlit Cloud。

- Effort: M
- Risk: Low
- Pros: 技术栈单一，部署最简单，现有经验可复用
- Cons: 录入体验不够顺滑（每次要打开网页填表），多用户session需要手写
- Reuses: 现有Streamlit页面模式、SQLite repository模式

### Approach B: 飞书Bot驱动 + Streamlit看板 (SELECTED)
飞书Bot接收格式化消息，正则解析入库。Streamlit做深度统计分析和预算管理。飞书用户OAuth自动识别身份。Streamlit网页录入作为非飞书成员的fallback。

- Effort: M-L
- Risk: Med
- Pros: 录入摩擦最低（聊天即记账），复用现有飞书集成代码，Bot推送报表到家庭群聊，体验最"wow"
- Cons: 开发量比纯Streamlit大，Bot消息解析需处理格式错误，需处理飞书事件订阅的边缘情况（去重、签名验证）
- Reuses: 现有飞书API集成、消息处理、auth模块

### Approach C: Gradio 替代 Streamlit
Gradio替代Streamlit，原生多用户auth，表单+图表组件丰富。部署到HuggingFace Spaces或自建。

- Effort: M
- Risk: Med
- Pros: 原生多用户auth（一行代码），Python技术栈一致
- Cons: 新框架学习成本，生态不如Streamlit成熟，部署灵活性不如Streamlit
- Reuses: Python后端逻辑

## Recommended Approach

**Approach B: 飞书Bot驱动 + Streamlit看板**

两个组件各司其职：
- **飞书Bot（FastAPI）**：消息接收 → 正则解析 → 入库 → 确认回复。处理飞书事件订阅（URL验证、签名校验、消息去重）
- **Streamlit**：统计分析 → 预算管理 → 费用类型配置 → 兜底网页录入

认证流程：
- **飞书用户**：点击Streamlit链接 → 飞书OAuth自动登录 → 识别身份
- **非飞书用户**：打开Streamlit → 下拉框选择"我是谁" → 输入4位PIN → 登录

Fallback设计：Streamlit保留简单录入表单（金额 + 类别 + 备注），供不使用飞书的家庭成员通过网页记账。

数据访问：Bot进程写入expenses，Streamlit进程读取+写入。SQLite WAL模式保证读写不互斥。

## Open Questions

1. 飞书Bot消息解析——MVP用正则严格格式，Phase 2评估是否需要LLM模糊解析
2. 家庭成员如果不都在同一个飞书租户内，Bot是否需处理外部联系人？
3. 导出功能（Excel/CSV）——大概率需要，作为Phase 2
4. 数据备份策略——SQLite文件定期cron备份到飞书云盘？
5. Streamlit OAuth回调的具体实现——需验证 st.query_params + session_state 方案的可行性

## Distribution Plan

- **飞书Bot**：部署为独立FastAPI服务，通过飞书开放平台事件订阅接收消息。部署到VPS，绑定飞书应用。
- **Streamlit**：部署到同一VPS的不同端口（如 :8501），反向代理到域名。
- **数据库**：SQLite文件放在VPS本地共享路径，WAL模式。cron定期备份。
- **部署方式**：git pull + systemd restart 两个服务。暂不引入CI/CD。
